## 第 2 页｜为什么选择电力设计院场景？

建议时间：35 秒

我选择电力设计院，一方面是因为这个场景与我接触过的 GIS、勘测和工程数据有一定关联；另一方面，它也非常适合展示数据从原始资料转化为数据产品的过程。

设计院的数据来源包括 GIS 平台、勘测成果、设计文件、工程文档、业务系统和外部数据。不同数据在格式、坐标、质量、安全等级和使用权限上存在明显差异，因此不能只做一个文件列表，而需要完整的接入、治理和资产化机制。

我的平台定位是，把这些分散数据逐步转化为可识别、可治理、可管理、可上架、可申请使用的数据产品。

底部这条链路可以概括为：分散数据经过标准化形成数据资产，再封装成数据产品，最终服务实际业务。

转场：确定场景以后，下一步是把面试题中的一句需求拆成产品方案。

## 第 3 页｜如何把一句需求转化为产品方案？

建议时间：40 秒

面试题的原始目标是：从外部数据接入，到形成数据资产目录，最终在数据资产门户上架数据。

我认为这里面还隐含了几个必要环节。数据接入后需要治理，否则无法判断数据是否规范、完整和可信；数据进入目录后，还需要完成质量、安全和权限检查，才能封装为数据产品；产品上架以后，还需要申请和审批，才能形成使用闭环。

因此我把需求拆成七个环节：数据源、数据接入、数据治理、数据资产目录、数据产品上架、数据产品门户，以及申请与审批。

这些环节分别回答：数据从哪里来、如何进入平台、是否可信、如何管理、如何产品化、如何被发现，以及如何获得使用授权。

这里体现的是，我没有把面试题理解成几个独立页面，而是先补齐了一条能够演示的业务闭环。

## 第 4 页｜平台整体由哪些层构成？

建议时间：45 秒

我把平台划分为五层。

最底层是数据源层，包括业务系统、GIS 平台、勘测成果、设计文件和外部数据。往上是数据接入层，负责数据库、API、文件、GIS、设计成果和实时数据等不同接入方式。

中间是数据治理层，处理标准化、质量检查、元数据完善和分类分级。治理后的结果进入数据资产层，形成资产目录、质量、标签、血缘和责任信息。

最上层是数据产品门户层，面向使用者提供检索、收藏、申请、审批和使用入口。

数据标准和安全权限是两项贯穿能力，它们会影响接入、治理、上架和使用全过程。这里表达的是目标架构设计，不代表当前前端 Demo 已经实现了生产级标准引擎或权限引擎。

在 Demo 中，这五层分别对应数据接入、数据治理、资产目录、产品上架、数据产品门户和个人工作台等页面。

## 第 5 页｜Demo 实际完成了哪些模块与交互？

建议时间：55 秒

这一页是 Demo 的功能矩阵，一共包含八个模块。

登录认证用于演示平台入口、路由保护和退出登录，登录状态保存在 sessionStorage。数据资产概览用于展示核心统计、快捷入口和数据资产建设流程。

数据产品门户面向数据使用者，支持产品搜索、筛选、详情、收藏和申请。建设端包含数据接入、数据治理、数据资产目录和数据产品上架四个核心模块。

数据接入展示多源接入配置、同步操作和日志；数据治理展示标准化、质量检查、元数据和分类分级；资产目录展示资产详情、标签、质量、血缘和上架申请；产品上架展示产品封装、质量安全检查、权限配置和上架流程。

个人工作台用于承接我的收藏、我的申请、待我审批和经手数据。

这里需要明确：这些模块已经具备可操作的前端交互，但业务数据和状态主要来自 Mock 数据及前端状态，不连接真实后端，也不会执行真实数据库同步或调度任务。

## 第 6 页｜一条电力 GIS 数据如何完成生命周期流转？

建议时间：50 秒

这一页以“广东省输电线路路径 GIS 数据集”作为主线案例。

它首先来源于广东输电线路 GIS 数据库。接入阶段需要配置数据源、字段信息和同步方式。

进入治理阶段后，平台模拟坐标系、行政区划、高程和字段规范等质量检查，同时补充元数据、目录分类、业务标签和安全等级。

治理完成后，数据进入资产目录，形成质量评分、责任信息、标签和血缘展示。资产发起上架申请后，在数据产品上架模块完成产品信息封装、质量检查、安全检查和权限配置。

上架以后，使用者可以在数据产品门户检索产品、查看详情并提交使用申请。申请进入个人工作台，再由管理员处理审批结果。

需要说明的是，当前 Demo 主要使用预置 Mock 状态，展示同一类数据在不同生命周期阶段的表现。部分操作存在轻量前端联动，但不是由真实后端驱动的同一条记录实时流转。

## 第 7 页｜如何用 AI 将模糊需求逐步落地？

建议时间：45 秒

这一页回答的是：我如何使用 AI，而不是一次性让 AI 生成几个页面。

我的过程分成五个阶段。首先进行需求调研，明确行业场景、数据对象和功能边界；然后进行方案与技术选型，比较前端框架、组件库和 AI 开发工具。

第三阶段是 Prompt 任务拆解，把一个大需求拆成平台骨架、页面功能、业务状态、Mock 数据、交互修正和视觉统一等可验证的小任务。

第四阶段由 Codex 分阶段实现，每次迭代只处理明确范围。第五阶段进行浏览器检查、lint、build 和 Git 版本管理，确保每次修改都可以验证和追溯。

我想强调的是，AI 负责辅助分析、生成和迭代；产品逻辑、功能取舍、业务边界和最终验收仍然由开发者负责。

## 第 8 页｜Prompt 迭代总览

PPT 左侧编号：08  
建议时间：30 秒

这一页先展示 Prompt 协作的整体方法。

两行分别对应“页面逻辑”和“视觉流程”两类任务。每一行都遵循同样的链路：左侧是我用自然语言描述目标，中间是 ChatGPT 进行结构化整理，右侧是 Codex 的执行与验证结果。

这里的关键不是让多个 AI 简单传话，而是逐步提高信息质量：先澄清真实意图，再补充范围、约束和验收标准，最后进入实现与验证。

底部对应项目的迭代过程：项目初始化、平台骨架、功能打通、交互修正、UI 统一和文档收尾。

所以我对 Prompt 的理解是，它不仅是生成指令，更是需求澄清、工程约束和质量控制的载体。

## 第 10 页｜视觉与流程 Prompt 示例

PPT 左侧编号：08  
建议时间：25 秒

这个例子聚焦数据治理与数据资产目录之间的流程一致性。

左侧自然语言中既有页面布局要求，也有治理任务继续处理、保存退出、提交目录、资产状态和操作前置条件等业务规则。

整理后的 Prompt 按 Git 与分支、总体业务规则、治理页面、资产目录、状态机、验证场景和提交汇报进行分层，避免只修改表面样式，却破坏业务状态。

例如，哪些状态允许编辑、哪些状态只能申请上架、治理异常如何退回处理，都需要先定义规则，再调整页面。

这个例子说明：视觉优化必须服从业务逻辑，Prompt 需要同时控制页面表现、状态约束和工程质量。

## 第 11 页｜为什么采用当前技术方案？边界在哪里？

PPT 左侧编号：09  
建议时间：50 秒

这一页回答两个问题：为什么采用当前技术方案，以及当前 Demo 做到哪里为止。

技术栈包括 React 19、Vite 8、Ant Design 6、React Router 7 和 JavaScript。数据和状态主要使用 Mock JSON、React 本地状态和 sessionStorage，工程协作使用 Git、GitHub 和 Codex。

设计取舍有四点。第一，Ant Design 更适合企业后台常见的表格、树、步骤和表单。第二，使用 Mock 数据，可以优先验证产品逻辑与交互闭环。第三，先打通流程，再统一优化 UI，可以降低返工成本。第四，当前规模有限，因此状态管理保持轻量。

当前边界同样需要明确：登录是前端 Mock；没有真实后端和数据库；状态主要保存在前端；审批和权限属于演示性实现。

如果进入生产环境，未来还需要补齐后端接口、持久化存储、数据接入调度、规则引擎、审计、真实权限体系和血缘计算等能力。这些都属于后续生产化工作，不是当前 Demo 已实现的功能。
