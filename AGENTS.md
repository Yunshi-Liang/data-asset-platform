# 电力设计院数据资产管理平台 Demo 开发规范

## 1. 项目背景

本项目是用于面试现场演示的“电力设计院数据资产管理平台”前端 Demo，面向电力设计院的数据资产管理场景，展示以下完整数据生命周期：

外部数据接入 → 数据治理 → 数据资产目录 → 数据产品上架 → 数据资产门户

项目重点展示产品设计能力、业务理解能力、前端交互能力和 AI 辅助开发能力，不开发真实后端服务。

### 1.1 产品与业务链路定位

- 数据资产门户是平台面向设计人员、项目管理人员和数据使用者的核心服务入口
- 平台后台处理链路统一为：数据接入 → 数据治理 → 数据资产目录 → 数据产品上架
- 平台业务背景聚焦华南地区，Mock 数据主要使用广东省、广西壮族自治区、海南省及其典型城市，不使用华东、华北、东北、西北等无关区域表达
- 主要业务域统一使用：技经数据、勘测数据、电气设计数据、工程数据、标准知识库

## 2. 技术栈

- React，使用 JavaScript，不使用 TypeScript
- Vite
- Ant Design
- `@ant-design/icons`
- React Router
- Mock JSON 数据，不接入真实后端
- 除非任务明确要求，不引入其他第三方框架或不必要的业务依赖

## 3. UI 与体验规范

- 整体采用蓝白科技风，参考国家电网、南方电网、Ant Design Pro 等企业级数据中台产品
- 布局简洁、专业，信息层级清晰
- 使用统一的色彩、间距、圆角、阴影、字体和交互反馈
- 突出关键指标、数据状态和数据可视化表达
- 页面应适合面试现场演示，优先保证核心流程清楚、视觉完成度高、操作反馈明确
- 避免无意义装饰、过度动画和信息堆叠
- 新页面与已有页面保持统一设计语言
- 公共布局沿用现有无顶部 Header 的沉浸式结构，用户信息和消息通知统一位于 Sider 底部
- 除非任务明确要求，不调整现有 Sider、导航顺序和公共布局

## 4. 开发原则

### 4.1 指令优先级与版本管理

- 最新 Prompt 的明确要求高于 AGENTS.md、既有约定和历史上下文；发生冲突时仅更新失效的长期规则，不无理由重写 AGENTS.md
- AGENTS.md 应保持简洁、明确，并作为仓库级长期开发规范提交到 Git
- 开始修改前检查 Git 工作区状态；完成独立任务后创建语义清晰的提交
- 未经用户确认，不执行强制推送、重写提交历史、删除远程分支、丢弃未提交修改等破坏性 Git 操作

- 采用任务驱动方式开发，只实现当前任务明确要求的功能，不提前开发后续模块
- 开始编码前先理解需求并说明实现方案
- 优先使用 React 与 Ant Design 的原生能力解决问题
- 使用本地 Mock JSON 表达业务数据，不实现真实接口或后端逻辑
- 不随意重构已经完成且运行正常的功能
- 修改范围应尽量聚焦，避免影响与当前任务无关的代码
- 当前开发阶段优先保证业务闭环、页面可操作和构建成功，暂不进行大规模视觉、文案和交互细节精修；核心模块完成后再统一优化
- 不删除或破坏已经可以正常运行的功能；确需调整公共代码时采用最小改动
- 重要交互需考虑加载态、空状态、错误提示和操作反馈
- 完成任务后说明修改文件、实现功能、验证结果和待确认事项
- 每项开发任务完成后执行 `npm.cmd run build`，确保项目可以正常构建
- Windows PowerShell 环境统一使用 `npm.cmd` 和 `npx.cmd`，不要修改 PowerShell 执行策略

## 5. 代码规范

- 使用 React 函数组件和 Hooks
- 文件使用 JavaScript/JSX，不创建 TypeScript 文件
- 组件化开发，页面、布局、通用组件、路由、Mock 数据、样式和工具函数按目录分类
- 组件、变量和函数命名应语义清晰，避免无意义缩写
- React 组件使用 PascalCase，普通变量和函数使用 camelCase
- 保持组件职责单一，避免超大组件和重复代码
- 公共逻辑或重复 UI 在确有复用需求时再抽取，不做过度设计
- 优先保证代码可读性；只在复杂业务规则、关键交互或不直观实现处添加必要注释
- 避免硬编码重复的颜色、间距和业务状态映射，统一配置可复用设计变量
- 保持导入顺序和代码格式一致，及时删除未使用的代码
- 不提交密钥、真实用户数据或敏感业务信息

## 6. 基础目录约定

```text
src/
├── assets/       # 图片、图标等静态资源
├── components/   # 可复用通用组件
├── layouts/      # 页面布局组件
├── pages/        # 业务页面
├── router/       # 路由配置
├── mock/         # Mock JSON 与模拟数据
├── styles/       # 全局样式、主题变量和公共样式
└── utils/        # 通用工具函数
```

## 7. 任务完成检查

每次开发完成后至少确认：

1. 当前任务要求是否全部实现，且没有提前扩展后续功能
2. 页面风格与现有设计语言是否一致
3. 是否存在明显控制台错误、未使用代码或重复实现
4. 路由、刷新和主要交互是否正常
5. 构建是否成功
6. 是否有需要产品或用户进一步确认的事项
